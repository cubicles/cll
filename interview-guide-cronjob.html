Backend Engineering — Interview Guide

The Cronjob
Reliability Questions

Three linked questions that move from diagnosis to implementation to scale and observability. Each is independent enough that a candidate can perform differently on each — and that gap is itself informative.

PHP / Python Junior · Mid · Senior ~60 min total Backend reliability
01 Diagnostic thinking
Question 1 — Open-ended, no right answer
"You have a cronjob that runs every night at 2am and has run fine for 8 months. This Monday it didn't run. Walk me through how you investigate — without looking at the code first."
open-ended no right answer ~10 min

The constraint "without looking at the code first" is the lever. It forces the candidate to reason about the system as a whole — infra, environment, external dependencies — before retreating to what's familiar. Listen for how they categorize their hypotheses, not just which hypothesis they pick first.

02 Implementation
Question 2 — Write it, make it safe
"Write a PHP script meant to run as a cronjob. It fetches a list of pending notifications from an external API, saves them to a local database, then processes and sends each one. Make sure it's production-safe: no duplicates if something crashes halfway, no double-runs if the previous execution is still going, and individual failures shouldn't bring the whole job down."
implementation PHP / Python ok 20–30 min

The fetch step is deliberate. It adds a third failure domain — the external API — on top of the database and the send. Most candidates write a safe happy path and only reveal gaps when you ask: "What if the API returns a 500 on page 3 of 5?" Their reaction to that follow-up is often more revealing than their original code.

03 Scale & observability
Question 3 — Design conversation, no code required
"Your script from Q2 now needs to process 5 million notifications a night instead of 5,000. And your team is tired of finding out about failures from angry users. What do you change — in the architecture and in the monitoring?"
system design observability ~20 min
Scale thread
The cronjob model breaks at volume. Listen for when they identify why — a single process, serial execution, no backpressure — before jumping to a solution.
Observability thread
"Tired of hearing from users" is intentionally vague. Strong candidates immediately ask what signals they're missing — not what tool to add.

This question has no single right architecture. What you're evaluating is whether they can reason about constraints — throughput, latency, cost, operational complexity — and make deliberate trade-offs rather than just naming technologies. A candidate who says "use Kafka" without explaining what problem Kafka solves here is weaker than one who says "a simple queue with N workers would handle this and be easier to operate."

Mid

Status transitions: pending → processing → sent / failed

Mid
Proposes parallel workers or a queue-based architecture with a clear reason why
Mentions alerting on completion time, not just failure — "if it hasn't finished by 4am, page someone"
Tracks a processed-count metric to catch silent degradation
Doesn't think about external API rate limits as a scale constraint
Monitoring covers the happy path only — no plan for partial failures
Senior
Identifies real bottlenecks: API rate limits, DB write throughput, send provider caps — not just CPU
Proposes a meaningful SLO: "95% of notifications sent within 2 hours of midnight" — not just "make it faster"
Considers cost: more workers = more API calls = more spend on the send provider
Monitoring includes a dead-man's switch — alert if the job hasn't started at all by 2:05am

Use these when a candidate raises a topic unprompted. The direction is usually a green flag — use the hint to calibrate depth, not to penalize the approach.

Retry handling any level ▼
Good sign at any level. Listen for whether they bound the retries — "retry forever" is a red flag, "retry 3 times with backoff then mark as failed" shows real production thinking. A senior should mention that retrying a send that may have already succeeded is different from retrying a fetch.
Follow-up: "What happens to the item after it exhausts its retries?"
Logging / observability expected mid+ ▼
Expected from mid+. A junior mentioning it unprompted is a strong green flag. Depth matters: logging "job ran" is table stakes. Logging item-level outcomes, durations, and external API response codes is mid. Emitting structured logs with a job run ID that ties every item back to one execution is senior.
Follow-up: "How would you know the job ran but processed zero items?"
SQS vs RabbitMQ mid+ signal ▼
Don't penalize for not knowing both. What matters is whether they understand why a queue helps here — decoupling fetch from send, built-in retry, natural backpressure. If they bring it up, ask what they'd lose by switching from a cronjob to a queue worker. A candidate who can name the trade-offs (operational complexity, visibility, local dev setup) is stronger than one who just says "queues are better."
Follow-up: "What would you lose by moving this to a queue worker?"
Dead letter queue senior signal ▼
Senior signal. It shows they've thought about items that fail repeatedly and need human review — not just automatic retry. If they mention it, ask: "who reads the dead letter queue, and when?" A good answer involves alerting and a manual reprocess path. A weak answer is "it just sits there." Bonus: do they distinguish between a DLQ for infrastructure failures vs a DLQ for business logic failures like malformed data?
Follow-up: "Who reads the dead letter queue, and when?"
The external API is down push this if they don't ▼
If they don't bring this up, ask it. It's the most realistic failure mode and tests whether they treat the fetch step as fallible — not just the send step. A junior might say "we catch the exception." A mid says "we log it and exit cleanly without touching the DB." A senior says "we exit cleanly, alert, and consider whether to retry the whole job or wait for the next scheduled run — and we track whether this is a recurring pattern."
Push: "What if the API returns a 500 on page 3 of 5?"
Horizontal scaling / parallel workers Q3 specific ▼
When they propose parallel workers, probe the coordination problem: how do two workers avoid picking up the same notification? If they haven't thought about it, they've just moved the duplicate-send problem from time (two sequential runs) to space (two simultaneous workers). A mid answers with DB-level row locking or a queue. A senior preemptively mentions it and explains why a queue makes worker coordination trivial.
Follow-up: "If you run 10 workers in parallel, how do they avoid processing the same item twice?"
SLOs / alerting thresholds Q3 specific — senior signal ▼
If they define an SLO unprompted — "99% of notifications delivered within X hours" — that's a strong senior signal. It means they think in terms of user impact, not just system health. Push further: "how do you alert before you breach the SLO, not after?" A good answer involves burn-rate alerts or a completion-rate curve — not just a binary success/failure check at the end of the job.
Follow-up: "How do you know you're going to miss your SLO at 3am, not at 6am when it's too late?"
