Backend Engineering — Interview Guide

The Cronjob
Reliability Questions

Two linked questions that probe diagnostic thinking and production-safe implementation. Independent enough that a candidate can perform differently on each — and that gap is informative.

PHP / Python Junior · Mid · Senior ~40 min total Backend reliability
01 Diagnostic thinking
Question 1 — Open-ended, no right answer
"You have a cronjob that runs every night at 2am and has run fine for 8 months. This Monday it didn't run. Walk me through how you investigate — without looking at the code first."
open-ended no right answer ~10 min

The constraint "without looking at the code first" is the lever. It forces the candidate to reason about the system as a whole — infra, environment, external dependencies — before retreating to what's familiar. Listen for how they categorize their hypotheses, not just which hypothesis they pick first.

02 Implementation
Question 2 — Write it, make it safe
"Write a PHP script meant to run as a cronjob. It fetches a list of pending notifications from an external API, saves them to a local database, then processes and sends each one. Make sure it's production-safe: no duplicates if something crashes halfway, no double-runs if the previous execution is still going, and individual failures shouldn't bring the whole job down."
implementation PHP / Python ok 20–30 min

The fetch step is deliberate. It adds a third failure domain — the external API — on top of the database and the send. Most candidates write a safe happy path and only reveal gaps when you ask: "What if the API returns a 500 on page 3 of 5?" Their reaction to that follow-up is often more revealing than their original code.

Senior
Recognizes absence of a log entry is itself a signal
Asks "why didn't we know until someone noticed?" — probes alerting gaps
Distinguishes job didn't start vs job started and silently failed
Thinks about blast radius — what broke downstream because this didn't run?
Jumps to solutions before finishing the diagnosis
Junior
Fetches from API, inserts to DB, sends — happy path works end to end
Wraps sends in try/catch so one failure doesn't stop the loop
No deduplication — re-running the script re-fetches and re-sends everything
No concurrency protection — two overlapping runs would double-send
Marks item as sent before confirming the send actually succeeded
Mid
Uses a lock file or DB mutex to prevent concurrent runs
Status transitions: pending → processing → sent / failed
Deduplicates fetched items by external ID before inserting
Logs job start, end, item count, and errors
Doesn't handle stuck jobs — items in "processing" from a previous crashed run
Lock file not cleaned up on crash — next run never starts
Senior
Stuck-job timeout: items in "processing" for N minutes get reset to pending
Idempotent send — marks sent only after confirmed delivery, handles the DB-update-fails-after-send edge case
Retry budget per item with backoff, not infinite retries

Use these when a candidate raises a topic unprompted. The direction is usually a green flag — use the hint to calibrate depth, not to penalize the approach.

Retry handling any level ▼
Good sign at any level. Listen for whether they bound the retries — "retry forever" is a red flag, "retry 3 times with backoff then mark as failed" shows real production thinking. A senior should mention that retrying a send that may have already succeeded is different from retrying a fetch.
Follow-up: "What happens to the item after it exhausts its retries?"
Logging / observability expected mid+ ▼
Expected from mid+. A junior mentioning it unprompted is a strong green flag. Depth matters: logging "job ran" is table stakes. Logging item-level outcomes, durations, and external API response codes is mid. Emitting structured logs with a job run ID that ties every item back to one execution is senior.
Follow-up: "How would you know the job ran but processed zero items?"
SQS vs RabbitMQ mid+ signal ▼
Don't penalize for not knowing both. What matters is whether they understand why a queue helps here — decoupling fetch from send, built-in retry, natural backpressure. If they bring it up, ask what they'd lose by switching from a cronjob to a queue worker. A candidate who can name the trade-offs (operational complexity, visibility, local dev setup) is stronger than one who just says "queues are better."
Follow-up: "What would you lose by moving this to a queue worker?"
Dead letter queue senior signal ▼
Senior signal. It shows they've thought about items that fail repeatedly and need human review — not just automatic retry. If they mention it, ask: "who reads the dead letter queue, and when?" A good answer involves alerting and a manual reprocess path. A weak answer is "it just sits there." Bonus: do they distinguish between a DLQ for infrastructure failures vs a DLQ for business logic failures (e.g. malformed data)?
Follow-up: "Who reads the dead letter queue, and when?"
The external API is down push this if they don't ▼
If they don't bring this up, ask it. It's the most realistic failure mode and tests whether they treat the fetch step as fallible — not just the send step. A junior might say "we catch the exception." A mid says "we log it and exit cleanly without touching the DB." A senior says "we exit cleanly, alert, and consider whether to retry the whole job or wait for the next scheduled run — and we track whether this is a recurring pattern."
Push: "What if the API returns a 500 on page 3 of 5?"
